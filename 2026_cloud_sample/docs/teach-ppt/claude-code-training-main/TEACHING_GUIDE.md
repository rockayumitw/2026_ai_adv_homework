## 單元 1：初探 Claude Code 特性與原理

教學設計——講師逐分鐘指引

**授課時長：** 40 分鐘
**授課對象：** 軟體工程師（已完成 Claude Code 安裝）
**教學形式：** 短講 + 即時 Demo + 學生跟做

### 教學目標

1. 學生能用終端機基本指令（cd、ls、mkdir）建立專案資料夾並啟動 Claude Code
2. 學生能操作 Claude Code 基本指令：/model、/clear、Ctrl+C、Escape、y/n 授權
3. 學生能請 Claude Code 建立 JS 檔案並執行，體驗完整的「對話 → 建檔 → 執行」流程
4. 學生能理解 Token 預測的概念，以及 Claude Code 的權限邊界

### 逐分鐘教學流程

| 時間 | 時長 | 階段 | 活動 |
|------|------|------|------|
| 0:00 - 3:00 | 3 min | 開場破冰 | 課程目標與環境確認 |
| 3:00 - 8:00 | 5 min | 終端機基礎 | 教 cd、ls、mkdir |
| 8:00 - 13:00 | 5 min | 啟動 Claude Code | 首次啟動與基本操作 |
| 13:00 - 18:00 | 5 min | 核心指令教學 | /model、/clear、Ctrl+C、Escape |
| 18:00 - 28:00 | 10 min | 核心實作 | ASCII 貓咪練習 |
| 28:00 - 35:00 | 7 min | 觀念講解 | Token 預測 + 能與不能 |
| 35:00 - 40:00 | 5 min | 收尾 | CLI 模式預告 + Q&A |

#### 0:00 - 3:00｜開場破冰（3 min）

**講師講解：**

「今天我們要學會用 AI 幫你寫程式。不是看 Demo，是你自己親手操作。」

快速確認：「大家的 Claude Code 都裝好了嗎？在終端機打 `claude --version` 有看到版本號的舉手。」

處理沒裝好的學員（助教協助），其他人先等一下。

#### 3:00 - 8:00｜終端機基礎（5 min）

**講師講解：**

「在開始之前，我們需要會三個終端機指令，再加一個小技巧。」

**講師示範：** 打開終端機，輸入 `ls` 看目前資料夾有什麼。

**講師示範：** 輸入 `mkdir claude-test` 建立一個新資料夾。

**講師示範：** 輸入 `cd claude-test` 進去這個資料夾。

「就這三個，記住：ls 看東西、mkdir 建資料夾、cd 移動。」

**講師補充 `&&`：** 「再教大家一個小技巧 —— 用 `&&` 可以把指令串起來一次跑。例如 `mkdir bmi-project && cd bmi-project && claude`，意思是建資料夾 → 進去 → 啟動 Claude，前一個成功才會跑下一個。後面的單元會常常用到。」

**學生跟做：** 給 30 秒讓學生跟著操作一遍。

> 💡 本單元結束會給一份完整的指令對照表，學生可以帶回家放著參考（包含終端機、claude 啟動、Claude 操作三大類）。

#### 8:00 - 13:00｜啟動 Claude Code（5 min）

**講師講解：**

「好，現在我們在 claude-test 資料夾裡了，輸入 `claude` 啟動。」

**講師示範：** 輸入 `claude`，等它啟動。

「它問你是否信任這個資料夾，選 Yes。因為我們剛自己建的，裡面是空的，很安全。」

介紹畫面：「這就是 Claude Code 的對話介面。你打字問它問題，它會回答你，還能幫你建檔案、寫程式。」

示範基本互動：打「你好，你是誰？」讓學生看到 Claude 回應。

「注意看，當它要建立檔案或執行指令的時候，會先問你 y 或 n。這是安全機制，它不會偷偷幫你做事。」

**學生跟做：** 啟動 `claude`，完成信任資料夾設定。

#### 13:00 - 18:00｜核心指令教學（5 min）

**講師講解：**

**講師示範 /model：**「輸入 `/model` 可以切換模型。目前有 Sonnet 跟 Opus，Sonnet 快但便宜、Opus 慢但聰明。平常用 Sonnet 就夠了。」

**講師示範 /clear：**「輸入 `/clear` 可以清除對話，重新開始。就像重新整理頁面一樣。」

**講師示範 Ctrl+C：**「如果 Claude 跑太久或你想中斷，按 Ctrl+C。」

**講師示範 Escape：**「打到一半不想送出？按 Escape 取消。」

「這四個操作記起來：/model 換模型、/clear 清對話、Ctrl+C 中斷、Escape 取消。日常 80% 的操作就靠這些。」

**學生跟做：** 讓學生各試一次 /model 和 /clear。

#### 18:00 - 28:00｜核心實作（10 min）

**講師講解：**

「好，現在來做一件有趣的事。請大家在 Claude Code 裡輸入：」

「幫我寫一個 cat.js，用 console.log 畫出一隻可愛的貓咪 ASCII art」

「它會幫你建立 cat.js 這個檔案，問你要不要允許，按 y。」

等學生都完成後：「現在開一個新的終端機視窗（或按 Ctrl+C 先離開 Claude），輸入 `node cat.js` 看看你的貓。」

「大家把自己的貓咪展示一下。」（可以投影或截圖分享）

**關鍵互動時刻：**

「有沒有發現？大家輸入的是一模一樣的指令，但每個人的貓咪都長不一樣。有人的貓在睡覺、有人的貓在站著、有人的比較胖。為什麼？」

（停頓，讓學生想一下）

**延伸練習：**「回到 Claude Code，跟它說『再加一隻狗在旁邊』或『讓貓咪戴帽子』，看看它怎麼修改。」

讓學生體驗「來回對話修改程式」的感覺，這就是 Claude Code 的核心互動模式。

#### 28:00 - 35:00｜觀念講解（7 min）

**講師講解：**

**講師解說 Token 預測：**

「剛才大家的貓咪都不一樣，原因是：LLM 不是從資料庫找一個標準答案。它是一個字一個字預測出來的。每一步都有很多可能的下一個字，不同的路徑就長出不同的貓。」

「想像你在寫一篇文章，每寫一個字，後面有很多種可能的接法。LLM 就是在做這件事，只是它每秒可以做幾千次。」

「這代表什麼？代表同樣的問題問兩次，答案可能不一樣。這不是 bug，是 LLM 的特性。」

**講師解說能與不能：**

「你們剛才有注意到，Claude 每次要建檔案、跑指令，都會先問你 y/n。這是設計好的安全機制。」

「Claude Code 能做的事：讀你的程式碼、寫新檔案、跑指令、幫你 debug。」

「不能做的事：它只能碰你信任的資料夾。它不會偷偷存取你的密碼、不會自己上網買東西、不會刪你的系統檔案。」

「記住一個原則：它是你的助手，不是你的老闆。每個重要動作都要你同意。」

#### 35:00 - 40:00｜收尾（5 min）

**講師講解：**

「最後教一個進階用法。除了進去對話模式，Claude 還能用單行指令：」

**講師示範：** 在終端機輸入 `claude -p "用一句話介紹 JavaScript"`

「這個 `-p` 模式不會進入對話，跑完就結束。以後你可以把這種指令寫在腳本裡，讓程式自動呼叫 Claude。下一堂課我們會用到。」

**課堂總結：**

「今天學了：終端機三指令、Claude Code 基本操作、寫了一隻 ASCII 貓、理解了 Token 預測。下堂課我們要學怎麼用更好的 Prompt 讓 Claude 寫出更精準的程式碼。」

**指令小抄（學生帶回家用）：**

| 類別 | 指令 | 用途 |
|------|------|------|
| 終端機 | `ls` | 列出目前資料夾的內容 |
| 終端機 | `mkdir <名稱>` | 建立資料夾 |
| 終端機 | `cd <名稱>` | 進入資料夾 |
| 終端機 | `A && B` | 前一個成功才跑下一個（串接指令） |
| 啟動 | `claude` | 在當前資料夾啟動 Claude Code |
| 啟動 | `claude -p "..."` | 單行模式，跑完就結束（適合腳本） |
| Claude | `/model` | 切換模型（Sonnet / Opus） |
| Claude | `/clear` | 清空對話、開新主題 |
| Claude | `Ctrl + C` | 中斷 Claude 正在做的事 |
| Claude | `Esc` | 取消正在輸入的訊息 |

開放 Q&A，回答學生問題。

- 準備助教協助處理安裝問題
- 測試教室網路能正常連線 Anthropic API

## 單元 2：CLAUDE.md 與專案設定

教學設計——講師逐分鐘指引

**授課時長：** 40 分鐘
**授課對象：** 軟體工程師（已完成單元 1）
**教學形式：** 實驗對照法（無設定 vs 有設定）+ 學生實作
**核心案例：** BMI 計算網頁

### 教學目標

1. 學生能理解 .claude 資料夾的架構與各檔案的用途
2. 學生能撰寫 CLAUDE.md 定義專案開發規範，並觀察到它對 Claude 產出的實際影響
3. 學生能設定 settings.json 控制 Claude 的權限，體驗放權後的流暢感
4. 學生能透過實驗對照，理解「設定檔決定產出品質」的核心觀念

### 教學設計理念

採用「實驗對照法」：學生先不做任何設定讓 Claude 產出 BMI 網頁（對照組），再加上 CLAUDE.md 重做一次（實驗組），透過前後對比讓學生親眼看到設定檔的影響力。最後加上 settings.json 放權，體驗流暢度的差異。

### 逐分鐘教學流程

| 時間 | 時長 | 階段 | 活動 |
|------|------|------|------|
| 0:00 - 2:00 | 2 min | 開場 | 回顧 + 今天目標 |
| 2:00 - 10:00 | 8 min | 對照組 | 不設定直接做 BMI 網頁 |
| 10:00 - 17:00 | 7 min | 認識 .claude | .claude 資料夾架構 + CLAUDE.md |
| 17:00 - 25:00 | 8 min | 實驗組 | 有 CLAUDE.md 重做 BMI 網頁 |
| 25:00 - 33:00 | 8 min | settings.json | 權限設定與放權 |
| 33:00 - 38:00 | 5 min | 成果展示 | 對比前後差異 + 討論 |
| 38:00 - 40:00 | 2 min | 收尾 | 總結 + 預告 |

#### 0:00 - 2:00｜開場（2 min）

**講師講解：**

「上次我們學會了基本操作，今天要學怎麼讓 Claude 『照你的規矩』寫程式。」

「我們會做一個 BMI 計算網頁，但重點不是網頁本身，而是你會親眼看到設定檔如何影響 Claude 的產出。」

#### 2:00 - 10:00｜對照組（8 min）

**講師講解：**

「首先，我們什麼都不設定，直接請 Claude 做一個 BMI 網頁。」

**講師示範：** 建立新資料夾並啟動 Claude。

`mkdir bmi-project && cd bmi-project && claude`

「在 Claude 裡輸入：」

「幫我做一個 BMI 計算網頁，用單一 HTML 檔案，包含 CSS 和 JS」

「按 y 允許建檔，等它寫完。」

**學生跟做：** 同樣步驟，建資料夾、啟動 Claude、請它做 BMI 網頁。

「好，用瀏覽器打開你的 HTML 檔看看成果。」

**關鍵提問：**「大家的網頁長得都不一樣對吧？有人深色、有人淺色、有人用中文、有人用英文。如果你想讓 Claude 每次都照你的風格來寫呢？」

「這就是接下來要教的東西。先記住你現在這個版本的樣子，等一下我們會對比。」

#### 10:00 - 17:00｜認識 .claude（7 min）

**講師講解：**

「現在我們來看一個重要的東西。在你的專案資料夾裡，有一個 .claude 資料夾。」

**講師示範：** 開新終端機視窗，輸入 `ls -la` 看到 .claude 資料夾。

`ls -la`
`ls .claude/`

「.claude 資料夾裡面有幾個重要東西：」

settings.json — 權限設定，控制 Claude 可以做什麼
CLAUDE.md — 專案規範，告訴 Claude 你的開發風格

「最重要的是 CLAUDE.md。Claude 每次啟動（開新對話）時會自動讀這個檔案**一次**，把內容載入 context，就像是你給新同事的一份『開發規範』。」

> ⚠️ **補充說明：** CLAUDE.md 只在開新對話時載入一次，之後就放在 context 裡。所以內容不要寫太長 —— 太長會吃掉 context 額度，後面對話容易被壓縮或失真。規範請寫得**精準、簡短**，把細節留給 Skill 或文件。

「現在我們來寫一份。」

**講師示範：** 在專案根目錄建立 CLAUDE.md。

「你可以用任何編輯器打開，或者直接請 Claude 幫你建。我示範一個簡單的內容：」

`# 專案開發規範`
`- 語言：繁體中文台灣用語`
`- CSS 風格：深色主題（背景 #0f172a）`
`- 所有函式要加中文註解`
`- 網頁要支援手機版面`

**學生實作：** 寫自己的 CLAUDE.md，至少寫 3-5 條規範。可以參考講師的，但鼓勵加入自己的偏好。

「寫完存檔就好，不用太多，3-5 條就夠了。」

#### 17:00 - 25:00｜實驗組（8 min）

**講師講解：**

「好，現在我們來驗證 CLAUDE.md 有沒有用。」

**講師示範：** 在 Claude 裡輸入 `/clear` 清除對話。

`/clear`

「輸入跟剛剛一模一樣的指令：」

「幫我做一個 BMI 計算網頁，用單一 HTML 檔案，包含 CSS 和 JS」

「注意看，同樣的指令，但這次 Claude 會先讀你的 CLAUDE.md。」

**學生跟做：** /clear 後重新下同樣的指令，讓 Claude 重做 BMI 網頁。

「用瀏覽器打開新的版本，跟剛剛的對比一下。」

**關鍵時刻：**「有沒有看到差異？如果你寫了『深色主題』，它這次就會是深色的。如果你寫了『繁體中文』，介面就會是中文的。」

「同樣的指令，不同的 CLAUDE.md，產出就不同。這就是 CLAUDE.md 的力量。」

#### 25:00 - 33:00｜settings.json（8 min）

**講師講解：**

「接下來教另一個重要的設定檔：settings.json。」

「你們剛剛有沒有覺得，每次 Claude 要建檔案都要按 y 很煩？如果你信任它，可以放權。」

**講師示範：** 打開 .claude/settings.json（如果不存在就建一個）。

「內容長這樣：」

`{ "permissions": { "allow": [ "Edit", "Write", "Bash(node:*)" ] } }`

「Edit 允許編輯檔案、Write 允許建立新檔案、Bash(node:*) 允許執行 node 指令。」

「設定完存檔，重新啟動 Claude。」

**學生跟做：** 建立或修改 .claude/settings.json，加入權限設定。

「現在請 Claude 幫你在 BMI 網頁加一個功能，例如『加一個歷史紀錄功能』或『加一個 BMI 分級表』。」

**關鍵體驗：**「有沒有發現？這次它直接就寫了，沒再問你 y/n。因為你已經在 settings.json 放權了。」

**安全提醒：**「放權很方便，但要注意範圍。不要寫 Bash(*) 全開，而是指定你需要的，例如 Bash(node:*) 只允許 node 相關指令。」

**補充：git 權限設定**

「順帶一提，Claude 也可以幫你執行 git 指令。如果你想讓它自動 commit，可以加：」

`"Bash(git add:*)"` `"Bash(git commit:*)"` `"Bash(git status)"` `"Bash(git diff:*)"`

「但注意，如果你不想讓 Claude 自動 push，就不要加 Bash(git push:*)。沒列的指令它就會先問你。這就是權限設計的精神：信任的給權限，不信任的保留確認。」

「這個今天不用實作，知道就好，下堂課會用到。」

#### 33:00 - 38:00｜成果展示（5 min）

**講師講解：**

「現在大家打開兩個版本的 BMI 網頁對比一下。」

「第一個版本：沒有任何設定，Claude 自由發揮。」

「第二個版本：有 CLAUDE.md 定義規範，Claude 照你的規矩來。」

**討論問題：**

「如果今天你進一家新公司，他們給你一份 coding guideline，那就是 CLAUDE.md 的概念。」

「而 settings.json 就像是給新同事的權限——你可以 push 到 dev branch，但不能直接 push 到 main。」

「這兩個檔案加在一起，就是你跟 Claude 協作的基礎。」

#### 38:00 - 40:00｜收尾（2 min）

**講師講解：**

**課堂總結：**

「今天學了三件事：」

「1. CLAUDE.md — 定義開發規範，Claude 每次都會讀」

「2. settings.json — 權限控制，決定 Claude 可以自動做什麼」

「3. .claude 資料夾 — 所有設定的家」

「下堂課我們要開始寫真正的程式碼，用 TDD 和 SDD 的方式讓 Claude 幫你開發。」

開放 Q&A。

## 單元 3：單元測試與 Plan Mode

教學設計——講師逐分鐘指引

**授課時長：** 40 分鐘
**授課對象：** 軟體工程師（已完成單元 2，有 BMI 專案）
**教學形式：** Plan Mode 規劃 + 單元測試實作 + Git 整合
**核心案例：** 延續單元 2 的 BMI 計算網頁

### 教學目標

1. 學生能使用 Plan Mode 讓 Claude 先規劃測試策略，並審核、修改計畫後再執行
2. 學生能幫現有程式補寫單元測試，體驗紅燈（測試失敗）→ 綠燈（測試通過）的流程，理解測試作為安全網的價值
3. 學生能使用 Claude 完成 git init → commit 的版本控制流程，串聯 settings.json 的權限設定
4. 學生能理解「規劃 → 測試 → 實作 → 提交」的專業開發循環

### 教學設計理念

延續單元 2 的 BMI 專案，學生不用從零開始，直接在已有的程式碼上補寫單元測試。透過「寫測試 → 看到紅燈 → 修改程式 → 看到綠燈」的親身體驗，讓學生理解單元測試是一張安全網——改了程式跑一下測試，全綠就知道沒改壞。搭配 Plan Mode 讓學生體驗「先想後做」，搭配 Git 整合讓學生走完一輪完整的專業開發流程。

「接下來我們用 Claude 的 Plan Mode 來規劃要測什麼。」

**講師示範：**

「輸入以下指令：」

**學生跟做：**

**`console.assert(1 === 1, '這行不會出現');`**

**`console.assert(1 === 2, '這行會標紅色錯誤');`**

「看到了嗎？第二行標紅了。這就是我們等一下要用的測試方法。」

「如果 Claude 建議安裝 Jest 或其他框架，跟它說：」

**講師講解：**

**Phase 1：先寫測試，看紅燈（5 min）**

「現在請 Claude 根據剛剛的計畫，先寫測試檔案。跟它說：」

**先建一個 bmi.test.html 測試頁面，用 console.assert 測試 BMI 計算函式，並把測試結果顯示在網頁上（綠色=通過、紅色=失敗）。先不要改 BMI 的程式碼**

等 Claude 寫完後：

「用瀏覽器打開 bmi.test.html 看看。」

「應該會看到一堆紅色的 FAIL。這就是紅燈——測試寫好了，但程式還沒準備好通過這些測試。」

「再按 F12 打開 Console，也能看到紅色的 Assertion failed。」

**關鍵時刻：**

「不要怕看到紅色！紅燈是好事，代表你的測試在工作。如果還沒改程式就全部通過，那測試才有問題。」

「好，現在跟 Claude 說：」

**測試跑失敗了，請修改 BMI 的程式碼讓所有測試通過**

Claude 會重構 BMI 的計算邏輯，可能會：

把 BMI 計算抽成獨立函式（如果還沒做的話）

加入錯誤處理

**關鍵時刻：**

「看到全部變綠色 PASS 了嗎？紅燈轉綠燈，代表你的程式通過所有測試了。」

「如果還有紅燈，把錯誤訊息截圖或複製貼給 Claude，讓它繼續修。這就是測試的循環：寫測試→看紅燈→修程式→看綠燈。」

**學生跟做：**

完成自己的紅燈→綠燈流程。助教巡場協助。

「全程只需要瀏覽器，不用裝任何東西。雙擊 HTML 就能跑測試。」

## 單元 4：Skills 封裝與團隊協作

教學設計——講師逐分鐘指引

**授課時長：** 40 分鐘
**授課對象：** 軟體工程師（已完成單元 1-3）
**教學形式：** 講師 Demo + 學生實作 + 展示交流
**核心案例：** 延續 BMI 專案，建立 /review 與 /commit 兩個 Skills

### 教學目標

1. 學生能理解 Skills 的用途：把重複性工作流程封裝成可重複使用的指令
2. 學生能建立 .claude/skills/ 資料夾結構，撰寫包含 Description、觸發條件、執行步驟的 SKILL.md
3. 學生能實作一個 code review skill 和一個 commit message skill，並實際驗證產出
4. 學生能理解 Skills 作為團隊知識資產的價值：標準化流程、降低溝通成本

### 教學設計理念

前三個單元讓學生學會了工具操作、規範設定、開發流程。單元 4 的定位是「封裝複用」——把前面學到的所有東西，濃縮成一個可重複使用的 skill 檔案。採用「講師帶做 + 學生獨立做」的雙軌設計：/review 由講師帶著做，確保學生理解完整流程；/commit 由學生自己做，驗證是否真正學會。最後的展示交流環節，讓學生看到同一個需求可以有不同的實作方式，激發回公司後自己寫 skill 的動力。

### 逐分鐘教學流程

| 0:00 - 2:00 | 2 min | 開場 | 回顧 + 今天目標 |
| 2:00 - 7:00 | 5 min | Skills 概念 | 結構、格式、觸發條件 |
| 7:00 - 20:00 | 13 min | 講師 Demo | 帶做 /review skill |
| 20:00 - 34:00 | 14 min | 學生實作 | 自己寫 /commit skill |
| 34:00 - 38:00 | 4 min | 展示交流 | 互看 skill + 討論 |
| 38:00 - 40:00 | 2 min | 收尾 | 四個單元總回顧 |

#### 0:00 - 2:00｜開場（2 min）

**講師講解：**

「我們花了三堂課，從零開始學會了 Claude Code。回顧一下你們會了什麼：」

「單元 1：啟動 Claude、基本指令、寫了一隻 ASCII 貓。」

「單元 2：用 CLAUDE.md 定規範、settings.json 放權，Claude 照你的規矩做事。」

「單元 3：Plan Mode 先規劃、幫程式補單元測試、完整開發流程。」

「今天是最後一堂，要教你們一個能帶回公司直接用的東西——Skills。」

「簡單說：你可以把常用的工作流程寫成一個檔案，以後一句話就能觸發整套 SOP。」

#### 2:00 - 7:00｜Skills 概念（5 min）

**講師講解：**

「Skills 放在專案的 .claude/skills/ 資料夾裡，每個 skill 是一個資料夾，裡面有一個 SKILL.md。」

**講師畫出結構：**

.claude/skills/
  review/
    SKILL.md
  commit/
    SKILL.md

「SKILL.md 就是一份 Markdown，告訴 Claude：什麼時候觸發、怎麼做、產出什麼。」

**講師說明 SKILL.md 的三個核心：**

「一份 SKILL.md 由兩段組成：最上面 `---` 包起來的 **frontmatter（metadata）**，下面是 **Markdown 內容**。重點看三個地方：」

「**1. `name`（必填）** — skill 的識別代號，只能用小寫字母、數字、連字號。例如 `code-review` ✅、`Code Review` ❌、`code_review` ❌。」

「**2. `description`（必填）** — 一句話描述 skill 做什麼，**這欄是 Claude 判斷要不要觸發的唯一依據**。要把『什麼時候用 + 用了會做什麼』講清楚，並包含使用者實際會講的關鍵字。」

「**3. Markdown 內容** — 實際的 SOP：審查重點、執行步驟、輸出格式...想得到的都可以寫。整份檔案建議 500 行內。」

**講師展示一份最小可用的 SKILL.md：**

````markdown
---
name: code-review
description: 當使用者要求進行 JavaScript 或 TypeScript 程式碼審查時使用。適用於「幫我 review」、「檢查這段程式碼」、「有什麼問題」等請求。
---

# JavaScript Code Review

當進行程式碼審查時，請依照以下步驟進行：

## 審查重點

1. **命名規範**：變數、函式命名是否清晰且符合 camelCase
2. **錯誤處理**：是否有適當的 try-catch 或錯誤邊界處理
3. **效能問題**：是否有不必要的迴圈、重複渲染或記憶體洩漏風險
4. **安全性**：是否有 XSS、注入攻擊等潛在風險
5. **可讀性**：程式碼是否易於理解，是否需要重構

## 輸出格式

請使用以下格式回覆：

### 🔴 必須修正
- [問題描述] → [建議做法]

### 🟡 建議改善
- [問題描述] → [建議做法]

### 🟢 優點
- [值得肯定的地方]
````

「重點看這三段：最上面 frontmatter 的 `description` 是 Claude 判斷要不要觸發的依據，寫得越精準越好；『何時觸發』補充情境；『執行步驟』就是 SOP。整份檔案不用很長，30 行內就能很實用。」

**⚠️ 講師強調 metadata 的重要性：**

「最上面 `---` 包起來的那段叫 **frontmatter**，也就是 metadata（後設資料）。它不是給人看的，是給 Claude 的『索引卡』。」

「Skill 不是你寫了 Claude 就一定會用。實際流程是：使用者講一句話 → Claude **只掃過所有 skill 的 description**（不讀內文）→ 覺得相關才會把整份 SKILL.md 載進來執行。」

「所以 `description` 等於 skill 的廣告詞，這欄沒寫好，skill 永遠不會被觸發。比較一下：」

- ❌ 爛 description：`description: code review`（Claude 不知道什麼情境該用）
- ✅ 好 description：`description: 當使用者要求 review、檢查、code review 程式碼品質時觸發，會依公司檢查清單逐項回報`（包含關鍵字、情境、產出，Claude 容易匹配）

「記住一個原則：description 要把『**什麼時候用 + 用了會做什麼**』講清楚，而且要包含使用者可能會講的關鍵字。」

**講師補充其他 metadata 細節：**

- **`name` 欄位規則**：只能用**小寫字母、數字、連字號**（`-`），不能有空格或大寫。例如 `code-review` ✅、`Code Review` ❌、`code_review` ❌。
- **檔案大小建議**：SKILL.md 維持在 **500 行以內**。太長會吃 context，也代表你應該拆成更小的 skill 或把細節放到外部腳本。
- **可選欄位**（進階）：
  - `allowed-tools`：限制這個 skill 可以用哪些工具
  - `user-invocable: true`：允許從斜線選單呼叫
  - `disable-model-invocation: true`：禁止 Claude 自動觸發，只能手動叫

**💡 重點觀念：Skill 呼叫腳本可以省 context**

「下個單元會講 Skill 串腳本。先告訴你一個關鍵：當 SKILL.md 裡叫 Claude 跑一個腳本時，**腳本的原始碼不會被載入 context**，只會把『執行結果』回傳給 Claude。」

「這代表什麼？你可以寫一個 1000 行的 Python 腳本，但 Claude 的 context 只多了幾行輸出。所以複雜邏輯都丟給腳本，SKILL.md 只負責『何時呼叫、怎麼解讀結果』，這才是省 token 又強大的寫法。」

**📚 延伸閱讀：** [Claude Code Skill 完整指南 - israynotarray](https://israynotarray.com/ai/20260120/4189755271/)

「它就是純文字檔，可以進版控。也就是說，你寫好一個 skill，團隊裡每個人的 Claude 都能用同一套流程。」

「想像一下：新人進公司，clone 專案，Claude 就自動有了你們團隊的 code review 標準、commit 規範。不用口頭教。」

#### 7:00 - 20:00｜講師 Demo（13 min）

**講師講解：**

「好，我們來實際做一個。先做一個 code review 的 skill。」

**Step 1：建立資料夾（講師示範）**

「在 bmi-project 裡，跟 Claude 說：」

**幫我建立 .claude/skills/review/SKILL.md，這是一個 code review 的 skill**

「讓 Claude 先幫你產一個初版。」

**Step 2：看 Claude 產出的 SKILL.md（2 min）**

「打開 SKILL.md 看看 Claude 寫了什麼。通常會有基本的結構，但我們要客製化。」

講師帶學生看 Description、觸發條件、執行步驟。

**Step 3：客製化 review 規則（5 min）**

「好的 code review skill 要有明確的檢查清單。跟 Claude 說：」

**修改 review skill，加入以下檢查項目：1. 函式是否超過 30 行 2. 有沒有 console.log 殘留 3. 變數命名是否語意化 4. 有沒有錯誤處理 5. 有沒有重複的程式碼**

「你可以根據你們公司的 coding guideline 來改。這就是 skill 的價值——把團隊標準寫進去。」

**Step 4：實際使用（3 min）**

「現在來測試。跟 Claude 說：」

**review 一下 BMI 的程式碼**

「看看 Claude 是不是照你的 skill 規則在做 review。」

**關鍵時刻：**

「有沒有看到？它不是隨便看一看，而是照你定義的五個項目逐一檢查。這就是 skill 跟直接問 Claude 的差別——有 SOP 的產出品質穩定、可預期。」

**學生跟做：**

跟著完成 review skill 的建立和測試。

#### 20:00 - 34:00｜學生實作（14 min）

**講師講解：**

「好，現在換你們自己寫一個 skill。我們來做一個自動產生 commit message 的 skill。」

**需求說明（2 min）：**

「這個 skill 要做的事：」

「1. 讀取目前有哪些檔案被修改了」

「2. 分析修改的內容」

「3. 根據修改內容，產生一個清楚的 commit message」

「4. commit message 要用繁體中文，格式是：類型（範圍）：描述。例如 feat(bmi)：新增歷史紀錄功能」

**學生動手（10 min）：**

「跟 Claude 說：」

**幫我建立 .claude/skills/commit/SKILL.md，這是一個自動產生 commit message 的 skill，規則如下：1. 讀取 git diff 分析修改內容 2. commit message 用繁體中文 3. 格式：類型（範圍）：描述 4. 類型包含 feat/fix/refactor/docs/test**

「Claude 產出後，打開 SKILL.md 看看內容，覺得要調整就跟 Claude 說。」

**驗證（2 min）：**

「隨便改 BMI 的一行程式碼，然後跟 Claude 說：」

**幫我產生這次修改的 commit message**

「看看它產出的 message 是不是照你的格式規範。如果不滿意，修改 SKILL.md 再試一次。」

「這個『寫 skill → 測試 → 調整』的循環，就是你以後優化 skill 的方式。」

#### 34:00 - 38:00｜展示交流（4 min）

**講師講解：**

「現在大家打開你的兩個 SKILL.md，跟旁邊的人交換看看。」

**討論問題：**

「1. 你的 review skill 檢查了哪些項目？有沒有別人寫了你沒想到的？」

「2. 你的 commit skill 格式規範是什麼？跟別人的有什麼不同？」

「3. 回公司後，你最想把哪個工作流程寫成 skill？」

「大家想想：如果你們團隊有 5 個人，每個人都用自己的方式 review、寫 commit message，品質會參差不齊。但如果把標準寫成 skill，放進專案裡，每個人的 Claude 都照同一套標準做事。」

「這就是 skill 的真正價值——不只是你個人的效率工具，而是團隊的知識資產。」

#### 38:00 - 40:00｜收尾（2 min）

**講師講解：**

**課堂總結：**

「四堂課，你們學會了完整的 Claude Code 工作流：」

「單元 1：認識工具 — 終端機操作、基本指令、Token 預測」

「單元 2：設定規範 — CLAUDE.md 定風格、settings.json 放權限」

「單元 3：專業流程 — Plan Mode 先規劃、補寫單元測試、完整開發循環」

「單元 4：封裝複用 — Skills 把 SOP 寫成可重複使用的指令」

「一條線串起來：從『讓 AI 隨便寫』→『讓 AI 照規矩寫』→『讓 AI 照流程寫』→『讓 AI 照 SOP 寫』。」

「你帶走的不只是工具的用法，而是一套跟 AI 協作的方法論。回去之後，遇到任何重複做的事，就想：這能不能寫成一個 skill？」

開放 Q&A，回答學生問題。

## 單元 5：綜合補充——實戰技巧與進階應用

教學設計——講師逐分鐘指引

**授課時長：** 40 分鐘
**授課對象：** 軟體工程師（已完成單元 1-4）
**教學形式：** 實戰技巧講解 + 講師展示（本單元無學生實作）
**核心案例：** Debug 工作流 + 講師真實專案中的 Skills 與 MCP 應用

### 教學目標

1. 學生能掌握 Debug 工作流：貼錯誤訊息、給重現步驟、讓 Claude 讀相關檔案
2. 學生能使用 Claude 快速理解陌生程式碼的架構與邏輯
3. 學生能理解 Context 管理原則：換主題清對話、一次一件事、規範寫進檔案
4. 學生能理解 Skill 可以呼叫自定義腳本，擴展 Claude 的能力邊界
5. 學生能理解 MCP 的概念：不用自己寫腳本，直接連接第三方服務

### 教學設計理念

本單元分三個區塊。第一區塊教三個日常實戰技巧（Debug、讀陌生程式碼、Context 管理），這些是工程師每天都會用到但前四個單元沒教的。第二區塊透過講師的真實專案案例，展示 Skill 從「純文字指引」到「串接腳本」的進階用法，涵蓋檔案產出、API 串接、瀏覽器自動化、排程推送、商業流程五種模式。第三區塊帶出 MCP 概念，讓學生知道除了自己寫腳本，還有現成的第三方整合可以用。

#### 0:00 - 2:00｜開場（2 min）

**講師講解：**

「前四堂課，你們已經學會了 Claude Code 的完整工作流：操作、設定、測試、Skills。」

「今天是綜合補充單元，分三個部分：先教三個日常一定會用到的實戰技巧，再展示 Skill 串腳本能做到什麼程度，最後介紹 MCP。」

#### 2:00 - 8:00｜實戰技巧 1（6 min）

**講師講解：**

「寫程式一定會遇到 bug，怎麼跟 Claude 協作 debug 最有效率？」

**錯誤示範：**

「最沒效率的方式：跟 Claude 說『壞了，幫我修』。它不知道壞在哪、怎麼壞的。」

**正確示範：**

「好的 debug 方式有三步：」

「1. 貼錯誤訊息 — 把完整的 error message 貼給 Claude，不要只說『有 bug』」

「2. 給重現步驟 — 告訴它『我做了 A → B → C，在 C 這步出錯』」

「3. 讓它讀相關檔案 — 跟它說『看一下 src/utils/bmi.js 的 calculateBMI 函式』」

**講師 Demo：**

用 BMI 專案示範：故意改壞一行程式，然後把錯誤訊息貼給 Claude。

「看，我只貼了錯誤訊息和發生的位置，Claude 就定位到問題了。」

「記住：你給的線索越多，Claude 修得越快。這跟你問同事 debug 是一樣的道理。」

#### 8:00 - 14:00｜實戰技巧 2（6 min）

**講師講解：**

「第二個技巧：用 Claude 快速看懂別人的程式碼。你換工作、接手舊專案一定會用到。」

**講師 Demo：**

打開一個學生沒看過的專案資料夾，啟動 Claude。

「跟 Claude 說：」

**幫我看一下這個專案的架構，主要的進入點在哪裡，用了哪些技術**

「Claude 會幫你掃過整個資料夾，告訴你：這是一個什麼專案、主要檔案有哪些、資料流怎麼走。」

「再深入一點，你可以問：」

**這個 calculateBMI 函式在做什麼？為什麼要除以 10000？**

「它會解釋函式邏輯，連那個看不懂的常數都會幫你解釋。」

「這招在接手舊專案的第一天特別好用。以前要花一週看懂的程式碼，現在可能一個下午就搞定。」

#### 14:00 - 19:00｜實戰技巧 3（5 min）

**講師講解：**

「最後一個技巧：Context 管理。這是用久了才會踩的坑。」

「Claude 有對話長度的上限。對話越長，它越容易忘記前面講的東西，回應速度也會變慢。」

**三個原則：**

「1. 換主題就 /clear — 你從 debug 切到寫新功能，先清對話。不要讓 Claude 帶著 debug 的 context 去寫新東西。」

「2. 一次一件事 — 不要在同一段對話裡又改 A 又改 B 又改 C。拆開來，每個任務一段對話。」

「3. 重要規範寫 CLAUDE.md — 不要每次對話都重複說『用繁體中文』『用深色主題』。寫進 CLAUDE.md，每次啟動自動讀取。這個你們在單元 2 就學過了。」

「簡單記：短對話、勤清除、規範寫進檔案。」

#### 19:00 - 21:00｜Skill 進階（2 min）

**講師講解：**

「接下來展示 Skill 的進階用法。」

「單元 4 你們寫的 /review、/commit，都是純文字指引。但 Skill 可以做更多——它可以叫 Claude 去跑你寫好的程式。」

「記住一句話：Skill 就是給 AI 的說明書。簡單的說明書教它怎麼 review，進階的說明書教它去呼叫你的腳本、串 API、操作瀏覽器。」

「接下來我快速展示六個真實案例，讓你們看看天花板在哪裡。」

#### 21:00 - 23:00｜展示 1（2 min）

**講師講解：**

「先問一個問題：Claude Code 為什麼能幫你產 Word 和 Excel？」

「答案是：它也是一個 Skill。Anthropic 官方寫了一個 Skill，背後呼叫 docx、xlsx 這些 npm 套件來產出檔案。」

**講師快速 Demo：**

**幫我產一份 Word 文件，標題是『會議紀錄』，內容隨便寫三條重點**

「模式就是：Skill 定義怎麼做 → Claude 去跑腳本 → 腳本用套件產出檔案。跟你們的 /review 同一套架構。」

#### 23:00 - 25:00｜展示 2（2 min）

**講師講解：**

「我自己寫了一個排行程的 Skill。SKILL.md 裡面叫 Claude 去跑我的 JS 腳本，腳本用 Google Calendar API 抓行事曆。」

**講師 Demo：**

「我說『這週哪天有空』，Claude 就去讀日曆，回覆哪些時段可以約。」

「Claude 本身不會連 Google 日曆，但你寫腳本幫它連，再用 Skill 告訴它什麼時候該跑。」

#### 25:00 - 27:00｜展示 3（2 min）

**講師講解：**

「同樣的模式。我有外縣市的課要上，需要規劃交通。」

**講師 Demo：**

「我說『下週三要去台北上課，早上十點到』，它幫我查高鐵班次、算轉乘時間、告訴我幾點出門。」

「Skill 是說明書，腳本是手腳。說明書定義要做什麼，腳本去查資料。」

#### 27:00 - 29:00｜展示 4（2 min）

**講師講解：**

「前面是串 API，接下來這個更猛——直接操作瀏覽器。」

「我有一個 Skill 可以登入財政部網站、自動繳費。背後用瀏覽器自動化，填表單、點按鈕。」

「付款確認還是我自己來，但登入、填表、查詢全部自動化。」

「只要是你在瀏覽器上重複做的操作，都能寫成 Skill + 腳本。登入公司後台撈報表、批次處理表單，都是同一個模式。」

#### 29:00 - 31:00｜展示 5（2 min）

**講師講解：**

「我每天早上收到一份早報，自動推到 Discord 頻道。」

「結合了排程（定時觸發）和推送（送到通訊平台）。」

「公司裡可以這樣用：每天跑部署報告推 Slack、每週彙整 commit 統計寄 email、監控 API 錯誤率超標就推通知。全部是 Skill + 腳本 + 排程。」

#### 31:00 - 33:00｜展示 6（2 min）

**講師講解：**

「最後一個。我跟 Claude 說『開一張發票，金額 5000，品項是教育訓練』，它自動產生財政部格式的 CSV。」

「商業流程自動化，不只是寫程式，連行政作業都能用同一套模式。」

「到這裡大家應該看到 pattern 了：不管是產文件、串 API、操作瀏覽器、推通知、處理行政——全部都是 Skill + 腳本。」

#### 33:00 - 38:00｜MCP 概念（5 min）

**講師講解：**

「前面六個展示，都是你自己寫腳本。但如果有人已經幫你寫好了呢？」

「這就是 MCP——Model Context Protocol。它讓 Claude 直接連上第三方服務，不用你自己寫腳本。」

**Canva MCP：**

「裝了 Canva MCP，你可以直接跟 Claude 說『幫我做一張活動海報』，它就會連上 Canva 幫你設計。不用寫任何腳本。」

**Figma MCP：**

「裝了 Figma MCP，Claude 就能讀取你的設計稿，甚至幫你從設計稿產出前端程式碼。」

「整理一下兩條路：」

「Skill + 腳本 = 自己造工具，完全客製，什麼都能做」

「MCP = 用別人造好的工具，裝了就能用，快速串接」

「社群上有很多現成的 MCP：GitHub、Slack、Notion、資料庫都有。有興趣的搜尋 MCP servers 去看看。」

#### 38:00 - 40:00｜收尾（2 min）

**講師講解：**

**課堂總結：**

「五堂課，我們走了一條完整的路：」

「回去之後，兩件事可以馬上做：1. 遇到 bug 先貼錯誤訊息給 Claude 2. 想想哪些重複動作可以寫成 Skill。」
